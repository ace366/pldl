# AGENTS.md

## 目的

このファイルは、このLaravelプロジェクトにおけるAI活用時の運用ルールを定義するためのものです。  
完全自動ではなく、**最初にAI判定を1回通す半自動方式** を前提とし、無理に即実装へ進まず、安全に進行ルートを決めてから作業を開始します。

---

## プロジェクト前提

- Framework: Laravel
- Editor: VSCode
- Source control: GitHub
- Build: local machine only
- Server: Sakura rental server

---

## 基本方針

- 既存コードを前提に、安全性を優先して変更する
- 実装前に、仕様の明確さと影響範囲を確認する
- 曖昧な依頼や影響範囲が広い依頼は、いきなり実装しない
- AIは役割を分けて使う
- **Claudeは分析・構成整理・影響確認を担当する**
- **Codexは実装・修正反映・コード変更を担当する**

---

## GitHub運用との整合

AI作業時も、既存の開発フローは崩さないこと。

- Issueを作成する
- Issueごとにブランチを切る
- Pull Requestで `main` にマージする
- PRには関連Issueを明記する

ブランチ命名の基本形:

- `feature/{issue番号}-{内容}`
- `fix/{issue番号}-{内容}`

コミット種別の基本形:

- `feat`: 機能追加
- `fix`: バグ修正
- `refactor`: リファクタ
- `chore`: その他

---

## AIの役割分担

### Claudeの役割

- 要件の読み解き
- 仕様の曖昧さの整理
- 影響範囲の洗い出し
- 関連画面、関連ロール、共通部品、DB変更有無の確認
- 将来復帰前提や切り戻し前提の対応整理
- 実装前の構成チェック

### Codexの役割

- 実装
- 既存コードの修正
- 必要ファイルの変更
- 局所的な不具合修正
- 分析済み内容に沿ったコード反映

---

## AI Routing Policy

修正依頼を受けたら、**いきなり実装せず、まず進行ルートを判定する** こと。  
この判定はスピードより安全性を優先し、迷う場合は **Claude先行** を選ぶ。

### 判定の基本

- まず修正内容を読み、対象画面・対象ロール・影響範囲・DB変更有無を確認する
- そのうえで、`Codex直行` と `Claude先行 → Codex実装` のどちらで進めるかを決める
- 不確実性が高い案件、横断的な案件、切り戻しが絡む案件はClaude先行とする

### Codex直行にする条件

以下に多く当てはまる場合は `Codex直行` とする。

- 対象画面が明確
- 対象ロールが明確
- 既存機能の延長
- DB変更がない、または軽微
- 影響範囲が局所的
- どのファイルを触るか概ね想像できる

### Claude先行にする条件

以下のいずれかが強く当てはまる場合は `Claude先行 → Codex実装` とする。

- 複数ロールに影響する
- `family` / `admin` / `staff` など複数画面にまたがる
- 既存機能の復活・停止・切り戻しがある
- DB変更の可能性がある
- 共通レイアウトや共通ナビに触る
- 既存仕様が曖昧
- 影響範囲が読みにくい
- 将来復帰前提の実装がある

### Routing Rule

- 軽微かつ局所的な修正は `Codex直行`
- 影響範囲が広い、または不確実な修正は `Claude先行 → Codex実装`
- 迷う場合は `Claude先行 → Codex実装` を優先する

---

## Required First Output

修正依頼に対する**最初の返答**では、必ず以下を出力すること。

1. 判定結果
   - `Codex直行`
   - `Claude先行 → Codex実装`
2. 判定理由
3. 想定影響範囲
4. 次に使うべきプロンプト

補足:

- この時点では、原則として実装を始めない
- まず判定と整理を返し、その後に適切なAIへ進める
- 安全に直行できると判断できた場合のみ、Codex実装へ進む

## triage-templateの利用方針

このプロジェクトでは、最初の判定に [prompts/shared/triage-template.md](/mnt/c/work/pldl/prompts/shared/triage-template.md) を使う。

- すべての修正依頼は、まず triage を通す
- 軽い修正は triage 後に `Codex直行` として実装へ進む
- 重い修正は triage 後に `Claude先行 → Codex実装` として進む
- triage結果は、そのまま次のAIに渡せる形に整理する

---

## 実務上の運用手順

1. 修正依頼を受ける
2. `prompts/shared/triage-template.md` で最初の判定を行う
3. `Codex直行` なら、整理済み内容を使ってCodexで実装する
4. `Claude先行 → Codex実装` なら、先にClaudeで構成チェックと影響整理を行う
5. Claudeで整理した内容をもとに、Codexで実装する
6. GitHub運用に沿ってブランチ、コミット、PRを進める

---

## 判定時の注意

- 「実装できそう」ではなく「安全に着手できるか」で判定する
- 共通部品、共通ナビ、認可、DB、複数ロールが絡む場合は慎重に扱う
- 一時停止機能、将来復帰機能、切り戻し案件はClaude先行を基本とする
- Laravelでは、画面だけでなくRoute、Controller、Blade、Request、Model、Service、Middleware、Migrationの影響も意識する

---

## まとめ

このプロジェクトのAI運用は、以下を基本とする。

- 最初に必ず進行ルートを判定する
- 軽微で局所的なら `Codex直行`
- 横断的、不確実、設計整理が必要なら `Claude先行 → Codex実装`
- 迷う場合は安全側で `Claude先行 → Codex実装`
- 実装前の入口として `prompts/shared/triage-template.md` を使う
